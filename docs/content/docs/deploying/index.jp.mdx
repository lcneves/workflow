Workflows は **ワールド** を通じて任意のインフラ上で実行できます。ワールドは、指定されたバックエンド経由でワークフローのストレージ、キューイング、認証、およびストリーミングを扱うアダプタです。

## ワールドとは何ですか？

**ワールド** はワークフローをそれらを動かすインフラストラクチャに接続します。ワークフローが存在し実行される「環境」と考えてください。ワールドのインターフェースはローカル開発と本番デプロイ間の違いを抽象化し、同じワークフローコードを異なる環境でシームレスに実行できるようにします。

## デフォルトの動作

ワールドはシナリオに応じて自動的に構成されます:

- **ローカル開発** - 自動的にローカル・ワールドを使用します
- **Vercel デプロイ** - 自動的に Vercel ワールドを使用します

他のワールドを使用する場合は、環境変数を通じて明示的に構成を設定できます。適切なワールドのドキュメントを参照して、構成の詳細を確認してください。

## ワールドを構築する

Workflow DevKit が提供するデフォルトのワールドに加えて、カスタムインフラ向けの新しいワールド実装を構築することもできます:

- データベースバックエンド（PostgreSQL、MySQL、MongoDB など）
- クラウドプロバイダ（AWS、GCP、Azure など）
- カスタムキューシステム
- サードパーティプラットフォーム

カスタムワールドを構築するには、コミュニティ実装の `World` を利用するか、`World` インターフェースを自分で実装してください。以下のインターフェースが必要です:

- **Storage** - ワークフローの実行、ステップ、フック、およびメタデータの永続化
- **Queue** - ワークフローステップを非同期にエンキューおよび処理する
- **AuthProvider** - API アクセスの認証処理
- **Streamer** - 読み取りおよび書き込み可能なストリームの管理

実装の詳細は [ワールド API リファレンス](/docs/deploying/world) を参照してください。

### サードパーティのワールドを使用する

カスタムバックエンドやサードパーティ製ワールド実装については、構成の詳細は各ワールドのドキュメントを参照してください。各ワールドは、必要な環境変数や構成オプションのセットが異なる場合があります。

## 可観測性

[可観測性ツール](/docs/observability)（CLI および Web UI）は、ワールドバックエンドに接続してワークフローデータを検査できます。デフォルトではローカル環境に接続しますが、リモート環境を検査するように構成することもできます:

```bash
# Inspect local workflows
npx workflow inspect runs

# Inspect remote workflows (custom worlds)
npx workflow inspect runs --backend <your-world-name>
```

[可観測性](/docs/observability) ツールの詳細を確認してください。
