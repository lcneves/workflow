## Crie seu projeto Nitro

Comece criando um novo projeto [Nitro v3](https://v3.nitro.build/). Este comando criará um novo diretório chamado `nitro-app` e configurará um projeto Nitro dentro dele.

```bash
npx create-nitro-app
```

Entre no diretório recém-criado:

```bash
cd nitro-app
```

### Instale `workflow`

```package-install
npm i workflow
```

### Configure o Nitro

Adicione o módulo `workflow/nitro` ao seu `nitro.config.ts`. Isso habilita o uso das diretivas `"use workflow"` e `"use step"`.

```typescript title="nitro.config.ts" lineNumbers
import { defineConfig } from "nitro";

export default defineConfig({
  serverDir: "./server",
  modules: ["workflow/nitro"],
});

```

## Crie seu primeiro fluxo de trabalho

Crie um novo arquivo para o nosso primeiro fluxo de trabalho:

```typescript title="workflows/user-signup.ts" lineNumbers
import { sleep } from "workflow";

export async function handleUserSignup(email: string) {
  "use workflow"; // [!code highlight]

  const user = await createUser(email);
  await sendWelcomeEmail(user);

  await sleep("5s"); // Pause for 5s - doesn't consume any resources
  await sendOnboardingEmail(user);

  console.log("Workflow is complete! Run 'npx workflow web' to inspect your run")

  return { userId: user.id, status: "onboarded" };
}
```

Vamos preencher essas funções que faltavam a seguir, mas antes dê uma olhada neste código:

- Definimos uma função de **fluxo de trabalho** com a diretiva `"use workflow"`. Pense na função de fluxo de trabalho como o _orquestrador_ das **etapas** individuais.
- A função `sleep` do Workflow DevKit nos permite suspender a execução do fluxo de trabalho sem consumir recursos. Um sleep pode ter duração de alguns segundos, horas, dias ou até meses.

## Crie as etapas do seu fluxo de trabalho

Agora vamos definir essas funções que faltavam.

```typescript title="workflows/user-signup.ts" lineNumbers
import { FatalError } from "workflow";

// Our workflow function defined earlier

async function createUser(email: string) {
  "use step"; // [!code highlight]

  console.log(`Creating user with email: ${email}`);

  // Full Node.js access - database calls, APIs, etc.
  return { id: crypto.randomUUID(), email };
}

async function sendWelcomeEmail(user: { id: string; email: string }) {
  "use step"; // [!code highlight]

  console.log(`Sending welcome email to user: ${user.id}`);

  if (Math.random() < 0.3) {
    // By default, steps will be retried for unhandled errors
    throw new Error("Retryable!");
  }
}

async function sendOnboardingEmail(user: { id: string; email: string }) {
  "use step"; // [!code highlight]

  if (!user.email.includes("@")) {
    // To skip retrying, throw a FatalError instead
    throw new FatalError("Invalid Email");
  }

  console.log(`Sending onboarding email to user: ${user.id}`);
}
```

Analisando este código:

- A lógica de negócio vive dentro das **etapas**. Quando uma etapa é invocada dentro de um **fluxo de trabalho**, ela é enfileirada para execução em uma requisição separada enquanto o fluxo de trabalho está suspenso, assim como `sleep`.
- Se uma etapa lançar um erro, como em `sendWelcomeEmail`, a etapa será automaticamente reexecutada até que obtenha sucesso (ou atinja o número máximo de tentativas da etapa).
- Etapas podem lançar um `FatalError` se um erro for intencional e não deva ser reexecutado.

## Crie seu manipulador de rota

Para invocar seu novo fluxo de trabalho, vamos criar um novo manipulador de rota de API em `server/api/signup.post.ts` com o seguinte código:

```typescript title="server/api/signup.post.ts"
import { start } from "workflow/api";
import { defineEventHandler } from "nitro/h3";
import { handleUserSignup } from "../../workflows/user-signup";

export default defineEventHandler(async ({ req }) => {
  const { email } = await req.json() as { email: string };
  // Executes asynchronously and doesn't block your app
  await start(handleUserSignup, [email]);
  return {
    message: "User signup workflow started",
  }
});
```

Este manipulador de rota cria um endpoint de requisição `POST` em `/api/signup` que acionará seu fluxo de trabalho.

<Callout>
  Fluxos de trabalho podem ser acionados a partir de rotas de API ou qualquer código do lado do servidor.
</Callout>

## Executar em desenvolvimento

Para iniciar seu servidor de desenvolvimento, execute o seguinte comando no terminal no diretório raiz do Nitro:

```bash
npm run dev
```

Depois que o servidor de desenvolvimento estiver em execução, você pode disparar seu fluxo de trabalho executando este comando no terminal:

```bash
curl -X POST --json '{"email":"hello@example.com"}' http://localhost:3000/api/signup
```

Verifique os logs do servidor de desenvolvimento do Nitro para ver a execução do seu fluxo de trabalho, bem como as etapas que estão sendo processadas.

Além disso, você pode usar a [CLI ou Web UI do Workflow DevKit](/docs/observability) para inspecionar as execuções do seu fluxo de trabalho e as etapas em detalhe.

```bash
# Open the observability Web UI
npx workflow web
# or if you prefer a terminal interface, use the CLI inspect command
npx workflow inspect runs
```
